# Evaluating Agent Context Quality
> [!zh]
> Agent 上下文（知识库）评估探讨

## 问题背景

在 AI Coding 落地过程中，团队通常会建设各种上下文能力，例如项目文档、仓库规则、组件库文档、基础库用例、业务知识、历史 bad case、RAG、MCP 工具、`AGENTS.md`、rules、skills 等。这些能力通常被统称为“知识库”或“上下文体系”。

但真正被挑战的问题不是“有没有知识库”，而是：

> - 什么样的上下文是好的？  
> - 什么样的知识库响应是有用的？  
> - 如何评价上下文建设质量？  
> - 如何用评价结果指导持续优化？

如果没有评价体系，知识库建设很容易变成材料堆积。上下文可能过时、重复、噪声大、不可执行，甚至因为干扰模型判断而降低任务效果。因此，Agent 上下文评估 的核心不是单纯扩大知识覆盖，而是建立一套可观测、可评价、可优化的上下文供应机制。

本次讨论聚焦最适合优先落地的一层：**知识调用质量（Knowledge Call Quality）**。

它要回答的问题很明确：

> 当主 Agent 已经发起某次知识查询时，知识系统是否返回了高质量上下文？

这里暂时不评价主 Agent 是否“应该查询某类知识”，也不评价最终任务是否成功，而是先把“知识库被调用时给得好不好”这件事独立评估出来。

## 上下文质量的四个观察层次

Agent 上下文质量可以拆成四个层次。这个分层的目的，是避免把知识库问题、Agent 决策问题、上下文使用问题和最终工程结果混在一起。

```text
知识调用质量（Knowledge Call Quality）
前提：主 Agent 已经发起查询。
问题：知识系统是否返回了好内容？
关注：返回内容是否相关、充分、新鲜、可执行、低噪声，以及调用成本是否合理。

上下文触发质量（Context Trigger Quality）
前提：给定一个任务或阶段。
问题：主 Agent 是否查了该查的知识？
关注：是否漏查重要上下文，是否过度查询无关上下文。

上下文使用质量（Context Usage Quality）
前提：知识已经返回。
问题：主 Agent 是否正确使用了这些知识？
关注：返回知识是否被实际使用，是否被误解，是否体现在 plan、代码或验证中。

任务结果对齐（Task Outcome Alignment）
前提：任务完成。
问题：上下文质量是否改善了工程结果？
关注：是否减少 API 误用、越界修改、review comment、安全遗漏和人工介入。
````

其中，**知识调用质量**是知识系统自身的评价；后面三层更接近 Agent 质量评价和工程结果评价。

如果主 Agent 根本没有查询组件库，导致使用了错误组件，这是上下文触发质量的问题，不是知识调用质量的问题。

如果知识库返回了正确组件示例，但主 Agent 没有使用，这是上下文使用质量的问题，不是知识调用质量的问题。

如果知识库返回质量不错，Agent 也使用了，但最终 PR 仍然因为业务理解错误被打回，这是任务结果或业务语义问题。

只有当主 Agent 已经发起查询，但知识系统返回空结果、旧文档、无关片段、冲突内容或不可执行信息时，才是知识调用质量的问题。

## 知识调用质量的核心定义

知识调用质量评价的是一次知识调用的响应质量，不评价 Agent 整体聪不聪明，而评价知识库、文件、MCP、RAG 在 Agent 已经发起知识请求后，是否返回了高质量上下文。

这使它可以独立回答一个实际问题：

这个项目中的 RAG / MCP / 文件知识库，在真实任务调用中的服务质量怎么样？

它不问：

- 主 Agent 是否应该查询某类知识
- 主 Agent 是否在正确阶段查询
- 主 Agent 是否完整理解需求
- 主 Agent 是否正确使用返回知识
- 最终 PR 是否成功

它只问：

- 主 Agent 发起查询后，知识系统是否返回了相关内容
- 返回内容是否足够支持下一步行动
- 返回内容是否新鲜、准确、无冲突
- 返回内容是否可执行
- 返回内容是否噪声过高
- 该知识源的响应质量是否稳定
- 该类 query 的失败模式是什么

## 为什么应该先做知识调用质量

知识调用质量是最值得优先落地的一层，因为它有清晰端点。知识库调用通常表现为 tool call、RAG query、MCP call、文件检索或搜索请求，这些调用有明确输入和输出，适合记录和评估。

它也更容易自动化。相比分析完整对话、完整 PR 或最终任务结果，评价一次知识响应更小、更稳定，也更适合使用 evaluator / sub-agent 进行结构化打分。

它还能快速产生健康指标。即使暂时不评价任务成功率，也可以知道哪些知识源经常返回空结果，哪些内容过时，哪些 query 噪声高，哪些文档缺少可执行信息。

更重要的是，它能直接指导知识库优化。一次低质量知识响应通常可以指向具体维护动作，例如调整 chunk、补充 metadata、优化 rerank、删除旧文档、改 MCP 返回结构、补充示例，或者将某些规则迁移到静态检查。

因此，知识调用质量是 Agent Context Quality 体系的第一期地基，而不是完整终局。

## 评估闭环

```text
1. 记录知识 tool call
   记录调用背景、query、source、返回内容、知识元信息、调用成本和后处理结果。

2. 对返回内容进行 sub-agent scoring
   评分维度包括 relevance、sufficiency、freshness、actionability、noise。

3. 聚合基础指标
   观察空结果、低质量调用、高成本低价值调用、过期内容、知识源质量和高频失败 query。

4. 抽样校准 evaluator
   定期人工抽样检查 sub-agent 评分是否可靠。

5. 沉淀 bad case
   将低分、高风险、高频失败的知识调用进入 bad case 池。

6. 回放验证优化
   修改 chunk、metadata、source 或 MCP 后，用历史低分 query 回放验证。
```

### 调用记录

第一期最重要的是先记录事实。即使暂时没有复杂评分，单纯记录调用次数、空结果比例、延迟、来源分布、过期比例、token 成本，也已经能提供知识系统健康信号。

这些日志是事实层，应该尽量机械、稳定、可复现。后续所有评分、指标、bad case 和优化动作，都应建立在这些事实记录上。

- 调用背景：记录这次知识调用发生在哪个任务、哪个阶段、由哪个 Agent 或工具触发。
- 查询信息：记录原始 query、改写后的 query、查询类型，以及查询消耗。
- 返回结果：记录返回了哪些知识片段，来自哪些知识源，排序和相关度如何，是否为空结果。
- 知识元信息：记录返回内容的来源、版本、更新时间、适用范围、是否可能过时或重复。
- 调用成本：记录延迟、token 成本、MCP / RAG 调用成本，以及是否出现重试。
- 后处理结果：记录 rerank、过滤、去重、降权等处理是否发生，以及原因。

### 响应评分

Sub-agent 的职责不是评估整个任务是否成功，而是评估这一次知识库响应是否对当前 query / 当前任务阶段有用。

Sub-agent scoring 不应被理解为绝对客观真理，而是一个自动化质量信号。它最好保留评分理由，方便后续抽样校准和 bad case 复盘。

- Relevance：返回内容是否与 query 和当前任务相关。
- Sufficiency：返回内容是否足以支持主 Agent 下一步行动。
- Freshness：返回内容是否新鲜，是否可能过时，是否来自废弃版本。
- Actionability：返回内容是否能转化为明确实现、验证、判断或约束。
- Noise：返回内容是否混入大量无关信息，是否可能干扰主 Agent。
- Conflict：返回内容之间是否冲突，或与当前项目规则冲突。
- Risk：返回内容是否可能误导主 Agent。

### 生成指标

指标应保持克制，主要分为事实指标、评分指标和衍生指标。

事实指标来自程序化日志，不依赖模型判断，适合作为知识系统的基础健康监控。它们关注知识调用是否稳定、是否有结果、是否成本过高、是否频繁返回过期或重复内容。

评分指标来自 evaluator / sub-agent，适合发现“返回了内容但质量不够”的问题。它们关注返回内容是否相关、充分、新鲜、可执行，是否噪声过高，是否存在冲突和误导风险。

衍生指标用于指导知识库维护优先级。它们可以帮助团队识别低质量调用比例、高成本低价值调用、知识源质量差异、高频失败 query 类型、主要上下文失败模式，以及需要优先维护或下线的知识源。基于这一目标，关注如下几个视角：

- 哪些知识源经常被调用？
- 哪些调用经常没有结果？
- 哪些返回内容相关性低？
- 哪些返回内容相关但不充分？
- 哪些知识源过时、重复或噪声高？
- 哪些调用成本高但价值低？
- 哪些 query 类型反复失败？

### 指导优化

知识调用质量可以直接指导知识库供给层的优化。

当返回内容不相关，通常说明 query rewrite、chunk 粒度、metadata、source 分类、rerank 或文档拆分存在问题。

当返回内容相关但不充分，通常说明文档缺少示例、反例、边界条件、操作步骤，或者必要知识分散在多个 source 中，没有被有效聚合。

当返回内容过时，通常说明知识源缺少版本、更新时间、owner、适用范围，或者旧文档没有及时下线。

当返回内容噪声过高，通常说明 top-k 过大、chunk 太长、metadata filter 不足、source 权重不合理，或者 query 本身过宽。

当返回内容不可执行，通常说明文档停留在背景介绍，缺少明确命令、代码示例、约束、do / don’t、checklist 或任务化说明。

当调用成本过高，通常说明 MCP 返回过重、检索链路过长、rerank 成本过高，或者高频知识没有缓存和前置。

这些问题对应的优化动作不只是“补文档”。有时应该删除旧文档、拆分 chunk、补充 metadata、调整 source 权重、优化 query template、改 MCP 返回结构、增加示例与反例；有时则应该把知识迁移为 skill、lint、CI、scope gate 或 contract check。

因此，知识调用质量评价不是为了让知识库越补越大，而是为了让知识系统更准确、更轻、更可执行。

## 价值与预期收益

知识调用质量的收益不应被夸大为“直接提升任务成功率多少”。它的直接收益是提升知识系统自身的健康度和可观测性；对最终 AI Coding 质量的影响是间接的，需要后续更高层的触发质量、使用质量和任务结果评价串联验证。

短期看，它能让知识调用变得可观测，让空召回、低相关、过期、噪声等问题被发现，让知识源质量可以被比较，并自动沉淀 context failure candidates。

中期看，它能推动 chunk、metadata、source、rerank、MCP 返回结构持续改善，让高价值知识源被更好维护，让低价值或过时知识源被下线或降权，并形成可回放的 bad case 集。

长期看，它能为上下文触发质量、上下文使用质量和任务结果评价提供数据基础，让知识库优化从主观感觉转向指标和 case 驱动。

## 风险与注意事项

不要把 sub-agent 分数当绝对真理。Sub-agent scoring 是自动化质量信号，不是人工专家判断的完全替代，应保留 rationale，并做抽样人工校准。

不要把知识调用质量结论外推到任务质量。它可以说某次知识响应质量差，不能直接说它导致任务失败。

不要让知识库越补越重。低分不一定意味着要补更多文档。很多时候应该删噪声、降权、工具化、规则化或静态检查化。

不要忽略知识来源与适用范围。没有 source、version、updated_at、owner、scope、applicable path 等 metadata，知识调用质量很难指导具体优化。

不要一开始做全量语义评价。第一阶段先做 tool call 层评价。Expected Context、完整对话分析、任务端到端评价留到后续阶段。

## 结论

Agent 上下文质量评价不应该一开始就试图评估整个 AI Coding 任务。更可落地的第一步，是从知识调用端点开始，评价知识系统在被调用时是否返回了高质量上下文。

知识调用质量的价值在于：它边界清晰、容易采集、适合自动化、能直接指导知识库优化、能沉淀 context bad case，并且能为后续更高层的上下文评价打基础。

它不能回答主 Agent 是否该查询、是否正确使用、最终任务是否成功，但它足以回答一个独立且重要的问题：

在真实 AI Coding 任务中，这个知识库 / RAG / MCP / 文件系统被调用时，给出的上下文质量到底好不好？

因此，知识调用质量是 Agent 上下文质量评估  体系中最适合优先落地的一层。它让知识库建设从“主观感觉有用”进入“可观测、可评分、可回放、可持续优化”的状态。
