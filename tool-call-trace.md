# Tool Call Trace：Agentic Coding 中一种值得验证的恢复上下文

## 1. 问题：中断后如何恢复探索过程

AI Coding Agent 的工作通常不只是一次性生成代码。对于稍复杂的任务，Agent 会搜索代码、读取文件、执行命令、查询文档、调用外部工具，并根据每次结果决定下一步行动。

因此，当 Agent 崩溃、上下文压缩、会话中断或用户主动暂停时，丢失的可能不只是“任务做到哪里了”，也包括一段已经发生过的探索过程。

Checklist 可以记录：

```text
当前任务尚未完成。
```

但它很难记录：

```text
此前查过哪些文件；
尝试过哪些命令；
哪些路径失败；
最近关注点在哪里。
```

因此，在 AI Coding 中，任务恢复可能不仅需要恢复任务状态，也需要恢复一部分探索上下文。

## 2. Tool Call 可能是高密度的探索记录

在 Agentic Coding 中，工具调用通常是 Agent 与环境交互的主要方式。一次搜索、一次文件读取、一次测试命令，表面上是工具调用，背后往往来自当前任务、已有上下文和中间判断。

例如：

```text
rg "refreshToken"
read src/auth/token.ts
read src/api/client.ts
npm test auth
```

这些记录不能证明 Agent 的判断正确，也不能完整还原它的推理过程。但它们至少表明：此前的 Agent 曾经围绕 `refreshToken` 搜索，读取过认证和 API client 相关文件，并运行过 auth 测试。

我的理解是，tool call trace 可能保存了一种被低估的信息：Agent 在任务上下文中已经外化出来的探索路径。

它不是结论，但它记录了：

```text
做过什么；
用什么参数做；
结果是什么；
后续行动如何变化。
```

这使它有可能成为一种比普通 task marker 更丰富的恢复上下文。

## 3. 为什么不直接保存完整过程

完整保存全部探索过程看起来更保真，但恢复时未必更有效。

完整过程通常包含大量内容：大段文件输出、重复搜索、测试日志、无关命令、中间尝试和被推翻的假设。这些内容当然有信息，但恢复时的单位 token 价值可能并不高。

恢复 Agent 通常更需要快速知道：

```text
最近在查什么；
查了哪些来源；
哪些命令失败；
哪些文件被改；
哪些路径可能相关。
```

因此，一个更轻量的方向是保存 tool call 的元数据，而不是完整过程本身。

也就是说，恢复上下文不一定追求完整，而应优先考虑单位 token 中的任务定位和行动线索密度。

## 4. 为什么不优先依赖智能摘要

另一种方式是让模型定期总结历史过程。但智能摘要会选择、压缩和解释历史记录，一旦摘要偏差进入持久层，后续 Agent 可能继承这个偏差。

例如，原始记录是：

```text
rg "refreshToken"
read src/auth/token.ts
npm test auth
```

摘要可能变成：

```text
问题集中在 refresh token 并发逻辑。
```

这个判断可能有帮助，但如果它尚未被验证，就可能在恢复时产生锚定。

相比之下，tool call trace 更接近原始事件。它不说“问题原因是 X”，只说“曾搜索 X，读取 Y，运行 Z，结果是什么”。恢复时的 Agent 仍然需要重新判断这些记录意味着什么。

因此，一个保守原则是：

```text
持久化行动和反馈，避免持久化未验证理解。
```

## 5. 局部窗口可能足够有用

这个方案还依赖另一个观察：Agent 的工具调用可能具有时间局部性。

当 Agent 围绕一个问题工作时，最近几十次工具调用通常更可能与当前任务相关。即使没有严格的 task id，最近窗口也可能提供恢复入口。

这类似排查系统日志时先看：

```bash
tail -n 50 log
```

不需要一开始读取全部历史，而是先看最近发生了什么，再决定是否扩大范围。

因此，极简方案不应把完整日志塞进上下文，而应提供局部读取能力：

```text
tail_tool_log(50)
tail_tool_log(100)
filter_tool_log({cwd, tool, status})
show_failed_tools(20)
```

如果读取全量日志，不同任务的高密度信息反而可能混在一起，增加 Agent 判断成本。

## 6. 极简实现

一个极简可行版本可以只包含两类纯程序能力。

第一，用 hooks 机械记录 tool call meta：

```text
时间；
工作目录；
工具名；
参数；
状态；
退出码；
响应 head / tail；
涉及文件；
搜索命中；
失败测试名。
```

第二，提供低成本读取工具：查看最近 50 条、最近 100 条，按目录、工具、失败状态或文件过滤。

恢复或排错时，让 Agent 先读取局部窗口，把它作为历史探索轨迹，而不是历史结论。必要时，再重新读取关键文件或重新运行关键命令。

## 7. 需要验证的两个假设

这个方案是否成立，取决于两个假设。

第一，tool call 参数和顺序是否具有足够高的信息密度。也就是说，Agent 搜什么、读什么、跑什么测试，是否确实能反映当前任务、已有上下文和中间判断。

第二，tool call 轨迹是否具有足够强的局部性。也就是说，最近一段工具调用是否通常围绕当前任务展开，使小窗口读取能够提供有效恢复入口。

如果这两个假设成立，tool call trace 可能会比单纯 task marker 提供更多恢复信息，也比智能摘要更少引入解释偏差。

但这需要实验验证，不能仅凭直觉确认。

## 8. 与社区实践的关系

社区中已经有接近的实践。Cline 保存 task history，并通过 checkpoints 支持任务恢复和代码回滚；SWE-agent 保存包含 action 与 observation 的 trajectory；OpenHands 使用事件流支持 conversation replay 和 state restoration。

这些实践说明，Agent 的行动轨迹已经被用于恢复、回放、调试和研究。

但目前较少看到一种明确的产品化方式：将 tool call meta 作为中断恢复和失败排查时的高密度证据链上下文，并让 Agent 通过局部窗口递进式读取。

这个方向可以视为完整会话、智能摘要和 task marker 之间的一种折中：

```text
完整会话：保真但臃肿；
智能摘要：紧凑但可能引入解释偏差；
task marker：简洁但信息不足；
tool call meta：低解释、较高密度、可局部读取。
```

## 9. 结论

Tool call trace 不应被视为完整 Agent Memory，也不应被视为推理状态恢复。它更像任务恢复和失败排查中的行动日志：不知道全部动机，但知道做过什么、先后顺序如何、世界如何响应。

一个值得验证的极简方向是：

```text
记录工具调用轨迹；
保留行动参数与环境反馈；
恢复时读取最近局部窗口；
让当前 Agent 重新判断下一步。
```

这个方案不能保证恢复成功，也不能证明旧判断正确。但如果 tool call trace 的信息密度和局部性假设成立，它可能显著降低 Agent 从中断、失败或上下文丢失状态重新进入任务的成本。
